# 开发计划：线性层正确性验证与参数配置动态化

本实现方案规定了在本科轮转期内，修正与验证本代码库中线性同态计算层的研究要求。方案主要分为两个流程阶段：首先量化现有流程的正确性基准，随后进行核心的参数重构工作。本方案不约束学生在面对算法维度限制时的具体实现代码框架。

## 零、环境搭建与 AI 辅助开发工具说明
本阶段需建立项目的基本运行环境并引入高效的开发辅助工具。

* [ ] **初始环境检查**
  * 配置并启用支持泛型的 Go 语言运行环境（版本需 >= 1.20）。
  * 在项目根目录执行 `go mod tidy` 拉取并构建项目依赖库（如 Lattigo 库）。
  * 取出项目内置测试参数检验原先的环境是否可用，参考验证命令：
    `go run main.go -test="Decoder" -hidDim=32 -seqLen=29`

* [ ] **开发工作流建议 (AI Coding)**
  * **背景说明**：鉴于全同态加密背景较为复杂及轮转周期（20天）的严格限制，研发的主要精力需集中于“逻辑设计、数据处理对齐及测试核验”，而非应对基础语法的组合与报错排查。
  * **实施建议**：强烈建议依托内置了 AI 代码生成功能的宿主编辑器（如 Cursor, Windsurf），或配合代码助手网站获取提示代码。
  * 对于撰写配置文件解析模版、编写数组比对算法或是遇到系统内核如越界报错时，皆可直接将控制台的日志与源文件供给大模型处理，生成基底代码后再加以组合验证。

## 第一阶段：自动化测试断言开发 (Encoding Correctness Validation)
建立正确的实验对比手段以提供后续阶段功能验收依据。

#### [MODIFY] main.go / [NEW] linear_test.go
* 构建基准测试流程：向常规明文算子路线 `LinearMsg()` 和加密通道路线 `Linear()` 输入一致的一组虚拟随机数据。运行完毕后将同态返回的密文层级数据执行代码解密操作以拉回到正常数组形态。
* 编写内置了系统均方误差 (MSE) 运算比对的测试闭环。确认所有结果数组中的各元素层的运算差别幅度维持在合理的极小量级误差范围内。
* 以目前自身便能跑通、且长宽被直接均匀整除的那些参数（如 `seqLen=29, hidDim=32` 等）开始起测。录入各运算块（如投影、QKV映射、UpGate、Out 层）计算精准性的输出误差统计结果作为性能验证锚点参照。

## 第二阶段：参数外置配置与维度兼容限制修复
移除因为非常规输入尺寸而在内部发生的报错，实现可随意配置并完成数据转换适配。

#### [MODIFY] main.go
* 开发与导入一组配置文件结构解析器，接管被硬编码写死的参数或老旧 `flag` 命令组。需要让它实现配置外部接管的功能，涵盖系统全局 `LlamaSize` 结构体内诸如 `seqLen`, `numHeads`, `hidDim`, `expDim` 等一切计算运行维度的动态调整赋值。

#### [MODIFY] util.go / linear.go (开放式开发路径)
* **技术限制说明**: 现存版本执行 CKKS 同态乘法操作时，要求原始长宽尺度可以满足预置处理的计算步长开方设定，一旦配置外置后传入非常规的结构大小（如偶数无法整开平方、质数行类、长宽异形的不对称配置），步长发生断裂就会引发内部报错停止。
* **开发约束**: 需针对异常断裂的问题制定措施。在不同的外部参数重设后，送入密文执行空间 `Linear()` 计算时都能妥善跑出且维持误差不出轨。
* **路径与自由度规划**: 执行思路解雇。你可以自由地在进入黑盒计算前通过进行矩阵的数据变换补齐 (Zero-Padding) 以满足内核大小而在完后截去长尾元素，间接地从工程层面弥补这一刚性缺陷；也可以直接潜入 `util.go` 或 `linear.go` 的底层计算封装代码中，改变原来步进划分与矩阵分配的多项式处理方法来寻求原生解决方案。

#### 测试与交接
* 在任务完毕后再度去调用第一阶段的那个 MSE 验证骨架框架，换上许多非常规、强异构的不成比例输入参数组。
* 当你们新产生的这部分实验数据输出误差依然趋于极小值平稳不崩溃时，即可出具结题的技术实验断言验收报告。
